SIEM 
 = Security Information and Event Management system
 - core security solution for a SOC analyst
 - collects all important logs in one centralized location
 - collects, normalizes & correlates them to identify incidents.

Log sources categories
 - Host-Centric
    - events that occurred within or related to the host
    - generated by Windows, Linux, servers, etc.
    - eg. user accesing a file, user authentication,
          process execution, powershell execution
 - Network-Centric
    - generated when hosts communicate with each other or access the internet
    - generated by firewalls, IDS/IPS, routers, switches, etc.
    - eg. ssh connection, ftp access to a file, 
          web traffic, firewall logs, IDS/IPS logs

Challenges that we will be solving with SIEM
 - large volume of logs and many different sources
 - no centralization
 - limited context (eg. a single log doesn't tell the whole story)
 - limited analysis (you can't do it manually)
 - format issues (requires skilled analysts to interpret the data)

SIEM
 - centralized log collection
 - normalizes them into a common format & size
 - correlates the logs to identify incidents (eg. a failed login followed by a successful login from the same IP is sus)
 - provides real-time monitoring and alerting for security events
 - provides reporting and dashboards for management and compliance (eg. Splunk dashboards)

Log sources and ingestion
 - on Windows we can use Event Viewer
 - on Linux we use the terminal; known log locations:
    - /var/log/syslog = system logs
    - /var/log/auth.log = authentication logs
    - /var/log/kern.log = kernel logs
    - var/log/httpd = web server access logs
    - /var/log/apache2/access.log = web server access logs
- Ingestion = give SIEM the logs
    - Agent / Forwarder = captures and sends important logs to SIEM
    - Syslog = standard protocol for sending logs over the network (UDP 514 default)
    - manually = uploading logs to SIEM (not recommended for larger environments)

Alerting process and analysis
 - SIEM has detection rules to catch threats (logical expressions)
    - eg. if failed login attempts > 5 in 10 minutes, 
          then raise alert for Multiple Failed Login Attempts
 - possible usecase
    - an event ID 104 is generated when a user tries to clear event logs
    - possible rule: If the Log source is WinEventLog AND EventID is 104 - Trigger an alert Event Log Cleared
 - we analyse the alert to determine if it's a true or false positive
    - case: True positive: investigate further
    - case: False positive: fine-tune the rule to avoid this in the future
